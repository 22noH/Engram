# Phase 12 — 다중 연결 (Multi-Connection)

작성일: 2026-07-07
상태: 설계 확정(브레인스토밍 산출) — 구현 플랜 대기

---

## 0. 이 문서의 위치 — 큰 그림 속 조각

Phase 11(클라이언트 앱 토대)에서 독립 React+Vite+TS 렌더러가 **로컬 두뇌 1개**에
ws로 붙는 구조를 지었다. Phase 12는 그 위에서 **여러 Engram에 동시 연결**하고,
연결마다 이름을 주고, `@`로 특정 Engram을 호출하는 단계다.

로드맵 분해표(§ phase11-client-app-design 0.2) 기준 위치:

| # | 서브프로젝트 | 이번 |
|---|------------|------|
| 11 | 클라이언트 앱 토대 | (완료) 로컬 1개 |
| **12 (이번)** | **다중 연결** | N개 연결, 연결 이름, `@`로 Engram 선택 |
| 13 | 인증 + 원격 노출 | ws 토큰 인증. "남의 두뇌 호출 금지"가 여기서 풀림 |
| 14 | 사람 팀 채팅 | 공용 대화 공간 실동작 |
| 15 | 원격/공유 Wiki 스토리지 | 여러 Engram 지식을 한 공간으로 |
| 16 | 다중 사용자/계정 | 동시 사용·격리·권한 |

**핵심 원칙(Phase 11 승계)**: 두뇌 코어는 손대지 않는다. 다중 연결·머지는 전부
**클라이언트(렌더러) 안**에서 끝난다. ws 프레임 계약도 변경 0 — 소켓 하나가 곧
Engram 하나라, 클라가 프레임을 로컬에서 연결 id로 태깅하면 된다.

### 0.1 용어

- **Engram Desktop** = 이 클라이언트 앱(사람이 쓰는 프로그램). 타이틀바 표기.
- **Engram** = 붙는 두뇌 하나(= 하나의 연결). 사용자가 이름(@tag)을 붙인다.
- **기본 Engram(default)** = `@` 없이 보낸 메시지가 가는 곳. 사용자가 지정/변경.
- **논리 채널(logical channel)** = 여러 Engram에 걸쳐 **이름으로 식별**되는 채널.
  화면엔 하나로 보이고, 실제 기록은 참여한 Engram들에 나뉘어 있다.

## 1. 이번 스코프

### 포함
- **다중 ws 연결**: N개 Engram에 각각 연결. 연결마다 `{ id, name, endpoint }`.
  연결 목록·기본값은 렌더러 **localStorage**에 저장(별도 설정파일·IPC 없음).
- **연결 관리 UI**("Manage Engrams" 모달): 추가(이름+endpoint)·삭제·기본 지정.
- **통일 라우팅 규칙**(모든 영역 동일): 메시지에 `@Engram이름` 있으면 그 Engram,
  없으면 기본 Engram으로.
- **기본 Engram 선택기**: 입력창 하단 **오른쪽**(보내기 버튼 옆) 칩 + 드롭다운.
  `/` 명령으로도 변경.
- **`@` 자동완성**: 입력 중 `@` 뒤에 연결된 Engram 이름 후보 표시.
- **공유 채널 + 스레드 머지**: 논리 채널의 기록은 그 이름 채널을 가진 모든 연결된
  Engram의 history를 **스레드(anchor) 시간순으로 합쳐** 보여준다.

### 비범위(후속)
- 토큰 인증·원격 노출(13) · 공용 대화 공간 실동작(14) · 공유 wiki 스토리지(15) ·
  계정/다중 사용자(16).
- **스레드 안에서 다른 Engram `@`호출**(교차): 이번엔 미지원(§4.3 규칙). 필요 시 확장.

### 안전 경계(명시)
Phase 12는 **인증이 없다**(13에서 추가). 그러므로 다중 연결은 **로컬 / 신뢰된 LAN
Engram**을 전제로 한다. 임의의 원격 Engram 연결은 13(토큰 인증) 전까지 "본인 책임 ·
로컬 개발용"이다. 이번에 인증을 짓지 않는 건 로드맵 순서 그대로이며, 설계에 명시만
해 둔다. endpoint 입력은 사용자가 직접 하고, 자동 탐색/브로드캐스트는 하지 않는다.

## 2. 아키텍처

### 2.1 단일 연결 → 다중 연결

지금(Phase 11): `useWs` 훅 1개가 `ws://127.0.0.1:<port>` 하나에 붙고, `App`이 그
한 두뇌의 channels/messages를 통째로 든다.

Phase 12: **연결 관리자**가 연결 목록을 들고, 연결마다 소켓 하나를 유지한다.

```
localStorage
  engram.connections = [{ id, name, endpoint }, ...]
  engram.defaultConnId = <id>
        │
   useConnections(연결 관리자 훅)
        │  연결마다: WebSocket + 재연결 백오프(1s→5s→30s) + 상태(connected)
        │  프레임 수신 시 connId를 붙여 상위로 (onFrame(connId, frame))
        ▼
   App(멀티 연결 인지)
        · 연결별 원시 상태: channelsByConn, msgsByConnCh (Engram별 채널 id·기록)
        · 파생(순수 머지): 논리 채널 목록, 논리 채널 통합 기록
        · 라우팅: (text, defaultConnId, connections) → targetConnId
```

**첫 실행 기본값**: 연결 목록이 비어 있으면 `{ name:'Local',
endpoint: ws://127.0.0.1:<현재 port> }` 하나를 시드하고 기본으로 지정. 즉 Phase 11
동작(로컬 1개)과 회귀 없이 동일하게 시작.

### 2.2 서버(두뇌) 변경 = 0

- ws 프레임 계약(`shared/protocol.ts`) 무변경. `createChannel`/`history`/`send`/
  `msg`/`channels`는 이미 있는 그대로 쓴다.
- 연결 id는 **와이어에 싣지 않는다**. 각 소켓이 하나의 Engram이므로 클라가 수신
  소켓으로 어느 Engram인지 안다 → 로컬에서 태깅.
- 두뇌는 자기가 받은 메시지·자기 답만 자기 `ChatStore`/개인위키에 저장(현행 그대로).

## 3. 논리 채널 + 스레드 머지

### 3.1 채널을 이름으로 식별

각 Engram은 자기 채널을 자기 id로 관리한다(서로 다른 id). 클라는 **채널 이름**을
논리 키로 삼아, 같은 이름의 채널을 하나의 논리 채널로 본다.

- **논리 채널 목록** = 연결된 모든 Engram의 채널 이름의 합집합(영역 `mode` 필터는
  현행 유지). `mode`/`respondMode`는 그 이름을 가진 채널들 중 대표값(첫 연결 or
  기본 Engram 것)을 쓴다.
- ponytail: 이름을 키로 쓰는 단순화. 서로 다른 Engram이 우연히 같은 이름의 무관한
  채널을 가지면 합쳐진다(허용 가능한 천장). 필요해지면 클라가 논리 채널 id를
  소유하고 Engram별 채널 id로 매핑하는 방식으로 승격.

### 3.2 통합 기록 = 스레드 머지

논리 채널을 열면, 클라는 그 이름 채널을 가진 **각 Engram의 history를 요청**하고,
받은 메시지를 합친다. 채널은 **스레드(anchor + 답) 묶음**이므로 머지는 스레드 단위다:

- anchor(비-threadId 메시지)들을 **ts 시간순 정렬**.
- 각 anchor의 답(threadId=anchor.id)은 그 anchor 아래 유지.
- 스레드 하나는 통째로 한 Engram에 있으므로(§3.3) 낱개 메시지가 Engram 간에
  뒤엉키지 않는다 — 머지는 anchor 정렬 + 답 부착이 전부.

수신 메시지에는 클라가 `connId`를 태깅해 둔다(어느 Engram의 스레드인지 = 답글
라우팅에 사용).

### 3.3 저장 범위 = "자기 것만"(addressed-only)

- 메시지는 **지목된(또는 기본) Engram에만** 보내 저장된다. 다른 Engram엔 뿌리지 않음.
- 결과: 각 Engram은 **자기가 참여한 스레드**만 갖는다. 통합 뷰는 클라가 합쳐 보여줄
  뿐, 어느 Engram도 전체를 갖지 않는다(정상 — 각자 독립된 제2의 두뇌).
- 지식 손실 없음: 채팅 기록(ChatStore)과 별개로, 각 Engram은 자기 턴을 자기
  개인위키(KnowledgeCore)에 저장한다. 여러 Engram 지식의 한-공간 통합은 Phase 15.

## 4. 라우팅

### 4.1 통일 규칙

```
target Engram =
  텍스트에 @<연결이름>이 있으면 → 그 Engram
  없으면                        → 기본 Engram(default)
```

Ask · Code · (미래) 공용 공간 어디서나 동일. 챗봇/코드에서 `@` 없이 보내면 기본
Engram이 답하고, `@회사`처럼 명시하면 그 Engram이 답한다.

### 4.2 전송 시 지연 생성

target Engram이 이 논리 채널(이름)에 해당하는 채널을 아직 안 가졌으면:

1. 그 Engram에 `{ t:'createChannel', name, mode }` 전송(지연 생성).
2. 그 Engram의 `channels` 브로드캐스트로 새 채널 id 확보 → 클라가 (connId, name)→id
   매핑 기록.
3. `{ t:'send', channelId:<그 Engram의 채널 id>, text, threadId? }` 전송.

"생각 중" 대기(`expectReply`)는 **논리 채널 단위**로 표시(현행 로직 재사용, 키만
논리 채널로).

### 4.3 스레드 안 답글 규칙

이미 열린 스레드에 답글을 달면, **그 스레드를 소유한 Engram**(anchor의 connId)으로
보낸다 — 답글 텍스트의 `@`는 이번엔 라우팅에 쓰지 않는다. 다른 Engram에 물으려면
**새 메시지(새 스레드)**로. (드문 경우라 이 기본으로 충분. 교차 스레드가 필요해지면
확장.)

### 4.4 채널 생성/삭제

- **생성**(UI에서 새 채널): **기본 Engram에만** 생성 → 그 Engram의 `channels`로 목록에
  등장. 다른 Engram엔 그 채널로 첫 메시지가 갈 때 §4.2로 지연 생성.
- **삭제**: 그 이름 채널을 가진 **모든 연결된 Engram에 `deleteChannel` 팬아웃**.
- **오프라인 Engram**: 팬아웃 대상에서 빠지고 재연결 시 반영 안 됨(천장). 이름 충돌·
  잔여 채널 정리는 필요해지면 후속.

## 5. UI 변화(최소)

### 5.1 기본 Engram 선택기 — 입력창 하단 오른쪽

- 입력바 아래 오른쪽(보내기 버튼 옆)에 칩: `[아이콘] <기본 Engram 이름> ▾` + 연결
  상태 점. 클릭 시 드롭다운:
  - 연결 목록(각 상태 점, 기본엔 ✓)
  - 구분선
  - **Manage Engrams…**(모달 열기)
- `/` 팔레트에도 기본 Engram 변경 항목 추가(연결 이름들).

### 5.2 Manage Engrams 모달

- 목록 행: 이름 · endpoint(mono) · [기본/기본으로] · 삭제.
- 추가 폼: 이름 입력 + endpoint 입력 + Add.
- 저장: localStorage(`engram.connections`, `engram.defaultConnId`). 변경 즉시
  연결 관리자에 반영(추가=연결 시작, 삭제=소켓 정리).

### 5.3 `@` 자동완성

- 입력값에서 커서 앞 토큰이 `@`로 시작하면, 연결된 Engram 이름 후보 팝오버.
  선택 시 `@이름 ` 삽입. 순수 필터(현 `/` 팔레트 패턴 재사용).

### 5.4 타이틀/문구

- 타이틀바: `Engram` → **`Engram Desktop`**.
- i18n(영어 기본 / ko 로케일): `Engrams`, `Add Engram`, `Manage Engrams`,
  `Default`, `@` 힌트 등 추가. 직역체 금지, 문장부호 규칙은 기존 톤 유지.

## 6. ws 프로토콜 / 공유 타입

- `shared/protocol.ts` **프레임 변경 없음**.
- 렌더러에 클라 전용 타입 추가(와이어 아님):
  ```ts
  interface Connection { id: string; name: string; endpoint: string }
  ```
- 연결 상태·(connId,name)→channelId 매핑은 렌더러 런타임 상태(저장 대상은
  Connection 목록 + defaultConnId뿐).

## 7. 에러 처리

- **연결별 독립**: 한 Engram이 끊겨도 나머지는 정상. 상태 점은 연결마다.
- **재연결**: 연결마다 백오프(1s→5s→30s) + 재연결 시 그 Engram 재동기화(현행 로직을
  연결 단위로). 통합 뷰는 남은 연결들 것만으로 계속 표시.
- **오프라인 Engram**: 그 Engram의 스레드는 재연결 전까지 통합 뷰에서 빠짐(허용).
- **기본 Engram이 오프라인인데 전송**: 소켓 미개방이면 전송 불가 → 에러 문구
  ("『회사』 연결 없음") 표시(무해 실패). 큐잉은 안 함(YAGNI, 필요 시 후속).
- **손상 프레임/미지 `t`**: 무시(현행).

## 8. 테스트 전략

렌더러(Vitest + Testing Library) 중심 — 서버 무변경이라 서버 테스트는 회귀만.

- **연결 관리자 훅**(모의 소켓): N개 연결 유지, 연결별 재연결 백오프, 수신 프레임의
  connId 태깅.
- **라우팅 순수 함수**: `(text, defaultConnId, connections) → targetConnId`
  (`@이름` 매치 / 미스 시 기본 / 대소문자·공백 경계).
- **스레드 머지 순수 함수**: Engram별 history 배열 → anchor ts 정렬 + 답 부착,
  낱개 뒤엉킴 없음.
- **논리 채널 목록 순수 함수**: Engram별 채널 → 이름 합집합(+mode 필터).
- **localStorage 지속**: connections/defaultConnId 저장·복원, 빈 목록 시 Local 시드.
- **UI**: 선택기 렌더·기본 표시·드롭다운, `@` 자동완성 삽입, Manage 모달 추가/삭제.
- **서버 회귀**: 프레임 계약 무변경 확인(기존 테스트 그대로 통과).
- Electron 실동작(다중 endpoint 연결)은 사용자 수동 스모크(기존 관례).

## 9. 마이그레이션·삭제

- 신규(렌더러): 연결 관리자 훅, 라우팅·머지·논리채널 순수 함수, Engram 선택기,
  Manage Engrams 모달, `@` 자동완성, localStorage 배선, i18n 문구.
- 변경(렌더러): `App`이 단일 `useWs` → 다중 연결 인지. 타이틀바 문구.
- 삭제: 없음(단일 `useWs`는 연결 관리자 내부로 흡수 or 연결당 재사용).
- 두뇌/서버·electron 빌드 설정: **무변경**.

## 10. 성공 기준

- Manage Engrams에서 두 번째 Engram(이름+endpoint)을 추가 → 두 연결 상태 점이 뜬다.
- 입력창 하단 오른쪽 선택기로 기본 Engram 변경(및 `/` 명령). `@이름`으로 특정 Engram
  호출.
- 한 논리 채널에서 서로 다른 Engram과의 스레드가 **시간순으로 섞여** 보이되, 스레드
  내부는 각자 온전(뒤엉킴 없음).
- 한 연결이 끊겨도 나머지 정상, 재연결 시 그 Engram 스레드 복귀.
- 첫 실행(연결 없음)은 Local 시드로 Phase 11과 동일 동작 — **회귀 0**.
- 두뇌 코어·ws 프레임 계약 **무변경**.
