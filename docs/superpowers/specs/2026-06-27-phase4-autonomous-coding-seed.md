# Phase 4 — 자율 코딩 협업 (씨앗 설계)

> 상위 기준선: [docs/DESIGN.md](../../DESIGN.md) §13.1 · 작성: 2026-06-27 · 상태: **씨앗 설계 — 방향 확정, Phase 3 seam 고정. 풀 구현 스펙 아님.**
> 전제: 이 능력은 Phase 3(Orchestrator 허브 + 이종/동종 두뇌 + 공유 위키 협업) 위에 얹힌다. **Phase 3가 아직 코드로 0이므로 여기서는 구현을 확정하지 않는다.**

---

## 1. 개요 · 범위

**궁극 목표**: 사람이 양 끝(처음 설계 + 마지막 확인)에만 있고, 중간 전부를 에이전트 무리가 서로 물고 돌려 *코드 산출물*을 자율 생성하는 팀. 협업은 협업대로, 코딩은 코딩대로 — Claude Code처럼 범용으로.

**이 문서의 깊이 (씨앗 설계 = depth A)**: 4대 미해결의 *방향*을 정하고, **Phase 3를 지을 때 깨면 안 되는 seam(이음매)** 을 못 박는다. 아직 없는 Phase 3 내부를 전제한 구현 스펙은 쓰지 않는다(과설계 방지). 풀 구현 스펙은 Phase 3가 실체가 된 뒤 별도 작성.

**범위 안**: 대상 범위 결정 / 코드 검증 게이트 방향 / 메시지 프로토콜 방향 / resume·제어 정책 방향 / Phase 3 seam 목록.

**범위 밖(미래)**: 자기 *소스코드* 수정·재배포(Phase 4 범위 제외, §2-③) · 풀 구현(클래스·파일·테스트) · C-2 특성화 자동생성의 사람 검토 게이트 · 이종 두뇌별 어댑터 세부.

## 2. 확정된 방향 (5)

| # | 미해결 | 결정 | 한 줄 근거 |
|---|---|---|---|
| ③ | 대상 범위 | **A — 외부 프로젝트만** | 자기파괴 위험을 통째로 회피하고 협업+검증 기계만 깨끗이 증명. 자기 코드 수정은 Phase 4 밖. |
| ① | 코드 검증 게이트 | **C — 2층** | 자동검사(하드 바닥) + 리뷰어 에이전트(소프트 위층). judge가 빨간 테스트를 못 덮음. |
| ④ | 메시지 프로토콜 | **B — 블랙보드 + 작업 티켓** | 에이전트는 서로를 호명 안 함. 공유 진실만 읽고 씀 → "둘이 같이 틀림"이 구조적으로 불가능. |
| ② | resume 정책 | **C′ — 진전·예산만 상한** | 시간/횟수 상한 없음(정상 장기작업 보호). "막힘"과 "오래 걸림"을 구분. |
| — | 제어(stop) | **run-state 게이트 하나** | stop·rate-limit·stuck·예산 네 트리거를 한 스위치로 통합. |

### ③ 대상 범위 = A (외부 프로젝트만)

Engram이 *사용자의 다른 repo / 새 프로젝트*를 짓는다. **자기 자신의 소스코드는 건드리지 않는다.** 자기개선(self-modification)은 무한루프·자기파괴 위험이 위키 갱신과 차원이 달라, 별도 안전장치가 필요 → Phase 4 범위 밖으로 명시 보류. 비전 전체(자기개선 포함)는 목적지로 두되, 씨앗은 안전한 땅에서 핵심 기계를 증명하는 데 집중.

### ① 코드 검증 게이트 = C (2층) + C-2

사람의 "오류 정정" 역할을 **게이트 통과한 객관 진실**이 대신한다. judge 에이전트가 "맞다"고 우겨도 코드는 진실이 못 된다 — 기계가 판정한다.

- **하드 바닥 (양보 불가)**: 프로젝트 자체 자동검사 — 테스트 통과 + 빌드 성공 + 타입체크/린트 그린. 리뷰어가 이 빨간불을 *초록으로 덮을 수 없다*.
- **소프트 위층**: 리뷰어 에이전트(작성자≠검증자, Phase 2의 writer≠judge 패턴)가 테스트가 못 잡는 설계·의도를 본다. *추가* 거부만 가능, *통과 강제* 불가.
- **테스트 없는 프로젝트 → C-2**: 에이전트가 코드를 바꾸기 전에 **특성화 테스트(characterization test)** 를 먼저 깐다 = 현재 동작을 *맞든 틀리든 그대로* 박제하는 그물. 목적은 정답 검증이 아니라 **회귀(의도치 않은 깨짐) 감지**.
- **새 프로젝트(greenfield)**: 박제할 기존 동작이 없음 → 사람이 처음 설계에서 준 **완성 조건(acceptance criteria)** 이 정답지. 에이전트가 그걸 테스트로 번역.

**알려진 한계(수용)**: C-2 특성화는 *기존 버그까지* 정답으로 박제한다. 그러나 Phase 4의 약속은 "**기존보다 나쁘게 만들지 않는다**"이지 "잠복 버그를 고친다"가 아니다. 버그는 *원래 있던* 문제이고, 자율 변경이 *새로 만든* 문제가 아니다. 따라서 수용.

### ④ 메시지 프로토콜 = B (블랙보드)

§3 불변(자유 채팅 금지 — 루프·비용 폭발)을 *구조적으로* 만족시킨다.

- 에이전트는 **다른 에이전트에게 절대 말하지 않는다.** 프로젝트-위키(SSOT)를 읽고 → 일 한 조각 하고 → 게이트 통과한 변경을 쓸 뿐.
- 유일한 "메시지" = **Orchestrator → 에이전트의 작업 티켓**("네 일은 이거, 현재 진실은 여기").
- "협업"은 *대화*가 아니라 **같은 칠판에 번갈아 쓰는 것**(stigmergy). 일이 검증된 일 위에 쌓인다. 협업은 두 층 — Orchestrator(조율: 분해·배정·종합) + 블랙보드(통합: 검증된 결과 위에 축적).
- **두뇌 roster는 brain-agnostic config.** Claude+Codex든 Claude+Claude든 같은 두뇌 N개든 동일하게 작동(`IBrainProvider` 포트가 추상화). 다양성은 모델/페르소나/검증독립성 중 어디서 와도 됨.

### ② resume 정책 = C′ (진전·예산만 상한)

rate-limit은 *고장*이 아니라 *리셋 시각이 정해진 알려진 throttle*. "실패"와 "예정된 대기"를 구분한다.

- **시간/재개횟수 상한 없음** — 큰 작업은 정상적으로 리셋 사이클 수십 번 걸친다. 시간 상한은 멀쩡한 장기작업을 죽인다.
- **상한 ① 진전 멈춤(stuck)**: "K번(2~3) 연속 재개했는데 칠판이 한 칸도 안 자람" → 사람 알림. 진전은 블랙보드로 객관 측정(게이트 통과 커밋·완성조건 미충족→충족). rate-limit 대기든 에이전트 루프든 한 신호로 잡는다.
- **상한 ② 예산(선택)**: 사용자가 처음 설계에서 정한 토큰/비용 천장. 시스템이 추측 못 함 — 사람이 정함. 안 정하면 진전 상한만 적용.
- rate-limit 대기 자체는 **진전하는 한 무제한**. → 비전("던지고 며칠 뒤 결과 확인")이 안 깨진다. 느려도 가면 안 깨우고, *멈췄을 때만* 깨운다.
- §10.2("모든 복구는 사람 알림에서 끝난다") 정신은 *비정상(stuck·예산초과)* 에 한해 보존.

### 제어(stop) = run-state 게이트 하나

사람이 "잠시 그만"을 걸 수 있어야 한다. 가능하며, B가 거저 풀어준다 — 게이트 통과한 것만 칠판에 올라가므로 칠판은 *항상 일관*, 아무 때나 멈춰도 잃을 게 없고 resume = 칠판 다시 읽고 이어서.

- 멈추는 지점 = **Orchestrator 배정구 단 하나**(허브 불변). "그만" = 다음 티켓을 안 냄.
- **Graceful**: 도는 작업 한 조각은 게이트까지 끝내고 그다음 티켓 보류(반쪽 상태 0).
- **Hard**: 도는 spawn 프로세스 즉시 종료. 미검증분은 칠판에 안 올라가니 그냥 폐기, resume 때 그 작업만 새로 재배정.
- 기본 = 일시정지(resume 가능). 강한 변형 = 취소(통째 폐기, resume 안 함).
- **통합**: stop·rate-limit·stuck·예산초과 네 트리거가 전부 같은 동작("배정 멈춤") → Orchestrator 배정구에 `run-state(running/paused/stopped)` 스위치 하나. 제어 명령(`engram pause/resume/stop`)은 Phase 1 CLI Gateway 위에 얹는다.

## 3. Phase 3가 지켜야 할 seam (씨앗의 산출물)

Phase 4를 도달 가능하게 유지하려면, Phase 3를 지을 때 아래를 깨지 말 것:

1. **Orchestrator는 유일한 배정구.** 모든 작업이 티켓으로 거기서 나간다. 에이전트 간 직접 호출/대화 경로를 만들지 말 것(§3 불변). → ④·제어가 여기 의존.
2. **에이전트는 stateless, 진실은 전부 SSOT에.** 작업 상태를 에이전트 프로세스/메모리에 들지 말 것. → resume·stop이 거저 되는 전제.
3. **Orchestrator 배정구에 run-state 스위치 자리.** Phase 3 동시성/턴 상한(세마포어·TurnBudget)을 만들 때 같은 지점에 pause/resume 게이트를 둘 수 있게 설계.
4. **진전(progress)은 블랙보드 변화로 관측 가능해야.** 게이트 통과 커밋·완성조건 상태가 칠판에서 객관적으로 읽혀야 stuck 감지가 성립.
5. **검증 게이트는 작성자≠검증자.** Phase 2의 writer≠judge 프로필 분리를 Phase 3 협업에도 유지. 코드 게이트의 소프트 층이 여기 올라탐.
6. **자기기만 차단**: 특성화/완성조건 테스트를 *바꾸려면* 그 변경이 사람의 완성 조건으로 추적돼야 한다. 에이전트 말만으론 테스트 갱신 불가.

## 4. 미해결로 남긴 것 (Phase 3 실체 후)

- 작업 티켓의 정확한 스키마(필드·SSOT 포인터 형식).
- run-state 스위치의 구현 위치(세마포어와의 결합 방식).
- 코드 게이트 "그린"의 프로젝트별 정의(테스트 러너·빌드 명령 탐지/설정).
- C-2 특성화 자동생성에 사람 흘끗 검토(약한 게이트)를 붙일지.
- 다중 에이전트가 같은 칠판에 동시 쓸 때의 락(Phase 0 `KeyedLock` 재사용 여부).
- self-modification(③) 해금 시의 추가 안전장치 설계(별도 Phase).

## 5. 한 줄 요약

> **외부 프로젝트를(③A), 블랙보드 경유로(④B) 협업하고, 자동검사 하드 게이트로 검증하며(①C), 진전이 멈출 때만 사람을 부르고(②C′), 한 스위치로 멈춘다(제어).** 풀 구현은 Phase 3 토대가 선 뒤.
