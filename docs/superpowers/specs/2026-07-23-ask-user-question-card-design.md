# 질문 카드(ask_user) — 두뇌가 선택지 UI로 묻기

날짜: 2026-07-23 · 상태: 설계 확정(목업 v4 사용자 승인)

## 목적

두뇌가 사용자에게 뭔가 물어야 할 때, 긴 텍스트 질문 대신 **선택지 카드**(옵션+설명+추천 배지+기타 자유입력)를 채팅에 게시한다. Claude Code의 AskUserQuestion과 같은 UX를 Engram 채팅 안에서 제공한다.

## 핵심 결정 (사용자 확정)

1. **비동기 모델**: 카드는 "버튼 달린 채팅 메시지"다. 두뇌는 카드를 게시하고 턴을 끝낸다. 사용자가 답하면(몇 시간 뒤라도) 그 답이 **새 사용자 메시지**로 전송되고 두뇌가 새 턴을 시작한다. 블로킹 대기 없음(재시작·타임아웃·무인 상황 안전).
2. **적용 범위 둘 다**: 자체 하네스(도구 루프) + 앱 MCP 노출(Claude CLI 하네스 채널도 커버).
3. **카드 요소 전부**: 옵션(라벨·설명·추천 배지)·기타 자유입력·Skip·다중선택·질문 묶음(1/N).

## UI (목업 v4 확정)

- **인라인 카드**: 채팅 스트림 안에 두뇌 메시지로 표시(오버레이 아님). 카드 상단 = ❓아이콘+질문 텍스트+(묶음이면 `1/N`).
- **옵션 행**: 맨 앞 **번호 칩**(선택/추천=accent 톤, 나머지 회색) + 라벨(굵게) + Recommended 배지 + 설명(회색). 추천 옵션은 테두리 accent.
- **기타 행**: 마지막 번호 칩 + 전체 폭 입력란(placeholder "Other — type your own answer").
- **하단 우측**: [Skip] [Send] 나란히. **클릭 = 선택 표시만, Send = 전송 확정**(단일·다중·기타·묶음 전부 동일 규칙 — 실수 클릭 즉시 전송 방지).
- **다중선택**: 옵션 행이 토글(체크 표시), Send로 확정.
- **질문 묶음**: 한 카드에서 차례로 진행. Send가 다음 질문으로 넘기고, 마지막 Send에 답들을 한 메시지로 합쳐 전송(`헤더: 답 / 헤더: 답`). Skip은 현재 질문만 건너뜀.
- **답한 뒤**: 카드가 answered 상태로 기록에 남음 — 선택 항목 강조(✓)·나머지 흐림·회색 톤·버튼 무동작(중복 전송 불가). 전부 Skip이면 skipped 상태.
- **키보드**: 카드에 포커스가 있을 때만 숫자키=옵션 선택, Enter=Send, Esc=포커스 해제. 채팅 입력창 타이핑 중엔 가로채지 않음.
- 문구는 영어 기본 + ko 로케일(기존 i18n 규칙). 접근성: 옵션은 실제 button/checkbox 시맨틱, 카드에 aria-label.

## 아키텍처

### 데이터 (A안: 메시지 구조 필드 — append-only 유지)

`shared/protocol.ts`의 `Message`에 additive 필드:

```ts
// 카드 메시지(두뇌 게시)
question?: {
  questions: Array<{
    q: string;            // 질문 텍스트
    header?: string;      // 짧은 라벨(묶음 답 합칠 때 접두)
    multiSelect?: boolean;
    options: Array<{ label: string; desc?: string; recommended?: boolean }>;
  }>;
}
// 답 메시지(사용자 전송) — 원본 카드 참조
answersId?: string; // question 메시지의 id
```

- **answered 판정은 참조로**: 카드 메시지를 수정하지 않는다. 렌더러가 채널 메시지 중 `answersId === 카드.id`인 메시지가 있으면 answered로 그린다(jsonl append-only 유지, 저장 update 불필요).
- **Skip 전체**도 답 메시지로 기록(`text: '(skipped)'` + answersId) — 카드가 영구 대기로 남지 않음.
- ⚠️ chat-store 로드 정규화 **필드 allow-list에 `question`·`answersId` 등재 필수**(미등재=저장 시 소실, 과거 함정).

### 흐름

1. 두뇌가 `ask_user` 도구 호출 → 서버가 question 필드 달린 두뇌 메시지를 채널에 append+브로드캐스트 → 도구는 "질문을 게시했음, 답은 다음 사용자 메시지로 도착함"을 반환 → 두뇌 턴 자연 종료.
2. 렌더러가 question 메시지를 카드로 렌더. Send 시 기존 send 경로 재사용 + `answersId` 첨부.
3. 서버 onSend: `answersId`가 있으면 해당 카드에 이미 답(같은 answersId 참조 메시지)이 존재하는지 검사 → 있으면 조용히 무시(중복 방지), 없으면 일반 메시지처럼 append+브로드캐스트+두뇌 트리거. 인증·authorId 스탬프 등 기존 게이트 전부 그대로.

### 두뇌 노출 경로 (플랜 조사로 확정 — MCP안 폐기)

조사 결과 Claude CLI 하네스는 매 메시지가 새 `claude -p` 프로세스이고 MCP 서버도 CLI가 직접 띄우는 외부 프로세스라, 앱 MCP에 "지금 이 채널" 정체성을 바인딩할 구조적 방법이 없다. 대신 두 경로로 같은 카드를 만든다:

- **① 자체 하네스 도구**: `ask_user` 도구(WebToolDef 관례) — input schema는 위 questions 구조의 축소형(질문 1~4개·옵션 2~4개 검증). 실행 클로저는 CompleteOpts로 주입(delegate 관례와 동일하게 orchestrator의 post 경로를 타고 내려옴).
- **② 응답 텍스트 후처리(범용)**: 어떤 하네스든 최종 응답에 ` ```ask_user ` 펜스 블록(JSON, 같은 questions 구조)을 넣으면 orchestrator가 게시 직전에 추출·검증해 카드로 게시하고 본문에서 블록을 제거한다. 검증 실패 시 블록을 건드리지 않고 그대로 텍스트로 둔다(오탐 안전). → **Claude CLI·도구 미지원 로컬 LLM까지 전부 커버**(MCP안보다 넓음).
- **프롬프트 지침**: TOOL_USAGE_GUIDANCE에 사용 지침+펜스 블록 형식 문서화 — "사용자 결정이 필요한 분기에서만, 옵션 2~4개, 추천 1개 표시. 예약·자동 트리거 등 무인 턴에서는 질문 남발 금지(답이 올 때까지 작업이 멈추므로)."
- **비-self 어댑터 폴백**: 카드 메시지의 `text`에는 질문+번호 옵션의 읽을 수 있는 텍스트 버전을 채운다 — question 필드를 모르는 표시면(예: 외부 메신저 어댑터)에서도 정보 유실 없음.

## 비목표

- 블로킹 대기·타임아웃·리마인더 없음(카드는 그냥 남는다).
- 웹 콘솔(/admin)·위키 영역 미적용 — 채팅 채널만.
- 기존 두뇌 텍스트 질문을 막지 않음(카드는 선택지가 있을 때만).

## 테스트

- chat-store: 필드 왕복(allow-list)·answersId 참조 보존.
- self.adapter: onSend의 answersId 경로 — 정상 답 append·중복 answersId 조용히 무시·question 카드 브로드캐스트·무인증 회귀 0.
- 도구: ask_user 스키마 검증(질문/옵션 개수)·메시지 생성 형태.
- 렌더러: 카드 렌더(번호칩·추천·1/N)·클릭=선택만·Send=전송·다중선택 토글·묶음 합침 포맷·answered 비활성·Skip 기록·키보드(포커스 시에만).
- 실스모크: 실서버+stub 두뇌로 카드 게시→답→두뇌 재트리거 왕복.

## 안전선

- 회귀 0: 프로토콜 additive·기존 send 게이트 재사용·question 없는 메시지 렌더 불변.
- 중복 전송 서버측 차단(클라 비활성은 UX일 뿐).
- 인증 모드에서 답 메시지도 기존 세션 게이트·authorId 서버 스탬프 그대로.
