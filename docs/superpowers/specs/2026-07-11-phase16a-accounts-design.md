# Phase 16a — 계정·신원 설계

작성일: 2026-07-11
상태: 설계 확정(브레인스토밍 산출) — 구현 플랜 대기

## 0. 이 문서의 위치 — Phase 16 분해

Phase 16("다중 사용자/계정 — 동시 사용·격리·권한")을 세 조각으로 분해. 이번은 첫 조각.

| # | 조각 | 내용 |
|---|------|------|
| **16a (이번)** | **계정·신원** | 1인 1계정(아이디/비밀번호 + OIDC SSO), 서버가 검증하는 작성자, 가입 승인제, 관리 화면 |
| 16b | 권한·역할 | 역할 세분화, 위키 승인권·하드삭제·게시관리 등 행위별 권한 |
| 16c | 격리 | 사용자별 위키 네임스페이스, 비공개 채널(`ownerId`/`visibility` 씨앗 활성화) |

**핵심 결정 — 운영 모델 전환.** 지금까지는 "로컬 무인증 + 원격은 공유 토큰 하나"(Phase 13)였다.
16a부터는 **앱은 서버에 로그인해야 쓸 수 있다**가 기본이다. 두뇌는 기본적으로 서버에 있고
(운영 주체가 회사든 팀이든 1인이든 구조 동일), 로컬 두뇌는 연산을 내 컴퓨터에서 돌리는
추가 옵션이다. 아직 배포된 적이 없으므로 마이그레이션 부담 없이 공유 토큰 방식을 제거하고
계정으로 대체한다.

## 1. 확정된 개념 모델 (사용자와 합의)

1. **앱 = 로그인 필수.** 앱을 켜면 첫 화면이 로그인. 예외 없음.
2. **두뇌는 서버에.** 회사가 운영하면 회사 서버, 팀이면 팀 서버, 혼자면 자기 컴퓨터의 1인
   서버 — 운영 주체만 다르고 구조는 같다.
3. **서버 주소는 배포판에 사전 설정.** SaaS가 아니므로 서버 운영자(회사 개발팀 등)가 자기
   서버 주소가 설정된 데스크톱 앱을 나눠준다. 팀원은 설치 → 로그인 → 사용.
4. **혼자 쓰는 사람** = 첫 실행에서 "내 서버 만들기"로 자기 컴퓨터에 서버를 세우고 첫
   계정(자동 owner)을 만들어 로그인. 회사가 쓰는 것과 같은 걸 자기 컴퓨터에 만드는 것.
5. **로컬 두뇌(`+`)** = 연산(LLM)을 서버 대신 내 컴퓨터에서 돌리는 선택지. 데이터 분리용이
   아니다 — 위키(지식)는 무조건 서버 한곳에 모인다(Phase 15 git 동기화가 통로).
6. **로그인 수단** = 아이디/비밀번호 + SSO(표준 OIDC — 구글 등 유명 제공자는 프리셋).
7. **가입은 소유자 승인제.** 셀프 가입·SSO 첫 로그인은 "대기" 상태로 들어오고 owner가
   승인해야 활성화.
8. **관리 화면은 앱 안에.** 별도 웹 대시보드 없음(Phase 11 원칙: 두뇌 http는 페이지를
   서빙하지 않는다). owner로 로그인한 사람에게만 관리 영역이 보인다.

## 2. 설계

### 2.1 실행 모드 — 서버 모드 / 두뇌 전용 모드

같은 런타임(데스크톱 앱에 이미 내장 — Phase 7 패키징)을 두 모드로 기동한다.
"똑같은 걸 통째로 또 설치"하지 않기 위한 분리다.

| | 서버 모드 | 두뇌 전용 모드 |
|---|---|---|
| 용도 | 팀/회사/1인 서버 | `+`로 추가한 로컬 두뇌 |
| 바인딩 | 개방 가능(원격 노출) | `127.0.0.1` 고정 |
| 계정·세션·로그인 창구 | 켬(필수) | 안 띄움(로컬 신뢰) |
| 팀채팅·위키 승인함·관리 프레임 | 켬 | 안 띄움(중앙 서버 몫) |
| Ask(두뇌에 질문) | 켬 | 켬 — 이게 존재 이유 |
| 위키 | 중앙 저장소 소유 | git 동기화로 중앙에 합류(15b/c) |

### 2.2 서버 쪽 모듈 (신규 — 단일 거대 클래스 금지)

기존 관례(ChatStore·WikiGit·page-merge가 각자 파일)를 따라 역할별로 쪼갠다.
`self.adapter`는 문지기 역할만 유지하고 계정 내부를 모른다.

- **`account-store.ts`** — `accounts.json` 읽기/쓰기 + 비밀번호 해시·검증. 순수 저장소,
  네트워크 모름. 계정 필드:
  - `id`(고정 식별자·uuid), `loginId`(로그인용·유일), `displayName`(표시이름),
    `pass`(scrypt 해시 + 사용자별 salt — 원문 저장 안 함),
    `oidc?`(`{ issuer, sub, email }` — SSO 연결 정보),
    `role`(`'owner' | 'member'` — 세분화는 16b), `status`(`'pending' | 'active' | 'suspended'`),
    `createdAt`.
  - 해시 = Node 내장 `crypto.scrypt`, 비교 = `crypto.timingSafeEqual`. 외부 라이브러리 없음.
- **`session-store.ts`** — 세션 발급·조회·만료·무효화. `sessions.json`:
  `{ token(crypto.randomBytes(32) hex), userId, createdAt, expiresAt }`. 만료 기본 30일,
  검사 시점 = 제시될 때. 사용자 단위 전체 무효화(`revokeAllFor(userId)`) 지원.
- **`auth-http.ts`** — `/auth/*` http 요청 파싱/응답만. 아래 2.3의 엔드포인트를 위 두
  저장소와 `oidc.ts`에 위임.
- **`oidc.ts`** — OIDC 절차(인가 URL 생성·콜백 코드 교환·id 토큰 검증)와 프리셋. 설정은
  서버 데이터 폴더 `auth.json`(issuer·clientId·clientSecret) — 관리 화면에서 수정 가능.
- **`self.adapter.ts`(기존)** — 변경은 두 곳뿐: ① http 요청이 `/auth/`면 `auth-http`로
  위임(한 줄 분기 — 헬스 프로브는 그대로) ② ws 인증 게이트가 공유 토큰 비교 대신
  `session-store`에 "이 토큰 누구냐"를 물어 소켓에 사용자를 매핑.

### 2.3 인증 프로토콜

**http (기존 헬스 프로브 서버에 추가):**

- `POST /auth/login` — `loginId`+`password` → 세션 토큰 + 사용자 정보. `pending`이면
  전용 오류("승인 대기 중"). 실패 응답은 균일 지연(무차별 대입 완화, 단순 고정 500ms).
- `POST /auth/register` — 가입 신청 → `pending` 계정 생성. `loginId` 중복은 오류.
- `POST /auth/logout` — 세션 무효화.
- `POST /auth/oidc/begin` — 구현은 리다이렉트 대신 JSON `{ authUrl, pollCode }`을 반환(데스크톱
  앱이 시스템 브라우저를 직접 열고 폴링). `state`에 일회용 폴링 코드 연계.
- `GET /auth/oidc/callback` — IdP가 돌아오는 곳. 코드 교환 → `issuer`+`sub`로 계정 조회,
  없으면 `pending` 계정 자동 생성(표시이름·이메일은 IdP 클레임에서) → 세션 발급 →
  브라우저에는 "앱으로 돌아가세요" 안내 페이지(이 응답 하나는 정적 텍스트 — UI 서빙이 아님).
- `GET /auth/oidc/poll?code=` — 데스크톱 앱이 폴링해 세션 토큰 수령(일회용, 짧은 TTL).
  원격 서버가 IdP에 등록된 콜백 URL을 소유하므로 데스크톱 루프백 리다이렉트 방식보다
  이 방식이 배포 그림(서버 주소 사전 설정)과 맞는다.

**ws (기존 프레임 재사용):**

- 기존 `{ t: 'auth', token }` 프레임의 `token` = 세션 토큰. 검증 성공 응답에
  `{ user: { id, displayName, role } }`을 실어 클라가 관리 영역 노출을 판단.
  `authErr`·5s 타임아웃·broadcast 게이트 등 Phase 13 상태머신은 그대로.
- **작성자는 서버가 찍는다.** 채팅 메시지·위키 제안의 작성자 = 소켓에 매핑된 세션의
  사용자. 클라이언트가 보낸 `authorId` 주장은 무시(Phase 14 자가선언 폐기). 메시지에는
  `authorId`(계정 `id` — 불변)와 `authorName`(그 시점 표시이름 — 렌더용)을 기록.
  `engram` 작성자 예약(사칭 방지)은 유지.

**강퇴·정지:** owner가 계정을 `suspended`로 바꾸면 그 사용자의 세션 전부 무효화 +
붙어 있는 소켓 즉시 종료.

### 2.4 첫 계정(서버 초기 설정)

계정이 0개인 서버 모드 = "미설정 상태". 기동 시 1회용 설정 코드를 생성해 콘솔/로그에
출력하고 서버 데이터 폴더에 `setup-code` 파일로도 남긴다. `POST /auth/setup`(설정 코드 +
loginId + password)으로 첫 계정을 `owner`·`active`로 생성하며, 성공 즉시 코드·파일을
지우고 이 엔드포인트는 닫힌다.

- **회사/headless 서버**: 관리자가 로그에서 코드를 읽어 자기 앱의 "서버 초기 설정" 화면에 입력.
- **혼자 쓰는 사람("내 서버 만들기")**: 앱이 로컬에 서버 모드를 띄우고 같은 흐름을 자동
  수행(`setup-code` 파일을 직접 읽어 주입) → 사용자는 아이디/비밀번호만 정하면 끝.

### 2.5 클라이언트(앱)

- **로그인 게이트**: 앱 시작 시 대상 서버에 유효한 저장 세션이 없으면 로그인 화면
  (아이디/비밀번호 + SSO 버튼 + 가입 신청). `pending` 계정이면 "소유자 승인 대기 중" 안내.
  세션은 로컬에 저장 — 매번 로그인하지 않는다.
- **첫 실행 분기**: 배포판에 서버 주소가 설정돼 있으면(설정 파일 동봉) 바로 그 서버 로그인
  화면. 없으면 "서버 주소 입력" 또는 "내 서버 만들기(이 컴퓨터)".
- **관리 영역(Admin)** — owner 세션에게만 표시. 통신은 전부 ws `admin` 프레임(15a
  `proposal` 프레임과 동형)이고 서버는 owner 세션에서 온 것만 처리:
  - 가입 승인함: `pending` 목록 + 승인/거부.
  - 멤버 목록: 상태 보기, 정지/복구, 비밀번호 리셋, 강제 로그아웃.
  - 서버 설정: 서버 이름, OIDC 설정(프리셋 선택 또는 issuer·clientId·clientSecret 입력).
- **Manage Engrams**: 토큰 입력칸 제거 → 연결별 "로그인" 흐름으로 대체. `+`(로컬 두뇌
  추가)는 두뇌 전용 모드로 기동·연결(로그인 불필요 — 2.1).
- **UI 문구는 영어 기본 / ko 로케일 한국어**(`i18n.ts` 관례).

### 2.6 기존 코드 정리 (배포 전 — 마이그레이션 불필요)

- Phase 13 공유 토큰 제거: `chat.config`의 `token`, `Connection.token`, Manage 토큰 입력칸,
  `main.ts`의 로컬 `?token=` 주입. 세션 방식으로 대체.
- Phase 14 자가선언 닉네임(`display-name.ts`, 닉네임 입력 UI) 제거 → 계정 `displayName` 사용.
- ws `auth` 프레임·`authErr`·인증 게이트 골격은 재사용(토큰의 의미만 교체).

## 3. 확장 한계와 경로 (정직하게 박제)

**16a의 목표 규모 = 개인 ~ 수백 명 팀.** 몇천 명급은 비범위.

- 계정·세션은 규모 문제가 아니다 — 로그인은 드문 이벤트고 데이터가 작아 JSON 파일로 충분.
- 몇천 명에서 먼저 터지는 곳은 **채팅 저장·브로드캐스트와 단일 프로세스**(Phase 9부터
  존재 — 16a가 만드는 병목이 아님). 경로: `ChatStore` 등 저장소가 모듈 경계 뒤에 있으므로
  필요 시점에 파일 → DB(SQLite → Postgres)로 그 모듈만 교체. 별도 Phase로 예약.
- **두뇌(LLM) 연산은 이미 수평 확장 가능** — 여러 두뇌가 한 위키에 합류하는 15c 구조 재사용.

## 4. 보안 노트 (신뢰 경계 — 단순화 금지 영역)

- 비밀번호: scrypt + 사용자별 salt, 비교는 `timingSafeEqual`. 원문·복호화 가능 저장 금지.
- 세션 토큰: `crypto.randomBytes(32)`. 로그·오류 메시지에 토큰 원문 노출 금지.
- 로그인 실패: 균일 응답(존재하지 않는 아이디와 오답을 구분 불가) + 고정 지연.
- OIDC: `state` 검증(CSRF), `id_token` 서명·`aud`·`iss` 검증, 폴링 코드는 일회용·짧은 TTL.
- `setup` 코드: 1회용, 첫 계정 생성 즉시 무효.
- 인증 전 소켓은 기존 게이트 그대로(비인증 프레임 무시·타임아웃 종료).
- TLS는 Phase 13 결정 유지 — 앞단(터널/프록시)의 몫, 문서로 안내.

## 5. 테스트 전략

- `account-store`/`session-store`: 순수 단위 테스트(해시 검증·만료·revoke·중복 loginId·
  손상 파일 fault-tolerant 로드 — 기존 store 관례).
- `auth-http`: 로그인 성공/실패/pending/setup 1회성/로그아웃 — http 핸들러 단위.
- `oidc`: 인가 URL 생성·state 검증·콜백 → pending 생성/기존 매칭 — IdP는 페이크.
- `self.adapter`: 세션 게이트(유효/무효/만료/suspended 즉시 끊김), authorId 서버 스탬프,
  admin 프레임 owner 전용.
- 렌더러: 로그인 게이트 분기·관리 영역 노출(role)·승인함 동작.
- 기존 공유 토큰 테스트는 세션 방식으로 치환. 나머지 스위트 무변경 통과가 회귀 기준.

## 6. 이번에 안 하는 것 (되살릴 신호)

- **행위별 권한 세분**(위키 승인권 분리 등) → 16b. 16a의 역할은 owner/member 둘뿐이고
  member 간 차등은 없다.
- **사용자별 위키 네임스페이스·비공개 채널** → 16c.
- **비밀번호 재설정 이메일** → owner가 관리 화면에서 직접 리셋. 메일 인프라 불필요.
- **2단계 인증(2FA)·프로필 사진** → 실사용 요구 시.
- **모니터링 대시보드(지표·그래프)** → 실운영 피드백 시. 16a 관리 화면은 계정 관리만.
- **DB 저장소 교체** → §3의 신호(실 대규모 배포) 발생 시 별도 Phase.
